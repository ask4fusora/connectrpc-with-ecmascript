# NestJS + Fastify Connect-RPC Backend

This directory contains a production-ready implementation of a Connect-RPC backend utilizing
**NestJS 10** and **Fastify v5**.

To ensure infinite horizontal scalability and maintainable code, this application strictly enforces
a **4-Pillar Feature-Driven Architecture**.

## The 4-Pillar Architecture

The codebase is mathematically bounded into four distinct layers. Rules regarding imports and
responsibilities across these boundaries are strictly enforced.

### 1. Entrypoint (`src/main.ts`, `src/app.module.ts`)

- **Responsibility:** The execution boundaries of the application.
- **Rules:** `main.ts` bootstraps the Fastify adapter and applies global middleware (e.g., CORS,
  disabling the native body parser). `app.module.ts` acts as the root dependency injection node,
  importing `core` and `features`. No business logic lives here.

### 2. Core (`src/core/`)

- **Responsibility:** Shared, generic, non-business infrastructure (e.g., the Connect-RPC Fastify
  plugin, IoC registries, logger configurations).
- **Rules:** Code in `core/` is completely agnostic to the application's domain. It must **never**
  import anything from `src/features/`. It provides generic interfaces (like `RpcHandler`) and
  registries that features bind to dynamically via Inversion of Control (IoC).

### 3. Features (`src/features/`)

- **Responsibility:** Pure business logic and domain-specific operations.
- **Rules:** This layer is highly cohesive. A feature module (e.g., `counter.module.ts`)
  encapsulates its own state (`counter.service.ts`) and its own network transport boundary
  (`counter.rpc-handler.ts`). It must not import from other features unless explicitly defined via
  an interface.

### 4. Gen (`src/gen/`)

- **Responsibility:** The Shared Kernel. Contains the deterministic API contracts generated by Buf
  (`@bufbuild/protobuf`).
- **Rules:** These files are strictly read-only. They are imported by `features` (to enforce type
  safety on business logic) and by `core` (to mount the Fastify schemas).

## Future Database Strategy (Drizzle ORM)

When introducing a database like Drizzle ORM, you must not create a "God folder" (e.g., `src/db/`
containing all schemas). The database integration must map perfectly into the existing 4 pillars:

1. **The Connection (Core):** The actual Postgres/MySQL connection pool and Drizzle initialization
   live in `src/core/database/database.module.ts`. This generic infrastructure provides the `db`
   instance via NestJS Dependency Injection.
2. **The Tables (Features):** Drizzle schemas belong inside the feature they represent. The
   `counters` table definition goes in `src/features/counter/counter.schema.ts`. If a feature is
   deleted, its API handler, business logic, and database table are cleanly deleted together.
3. **Migrations (Root):** Database migrations and `drizzle.config.ts` operate independently of the
   NestJS compiler. They must live at the root of the backend project, entirely outside of `src/`.

### Target Folder Structure

```text
backend/nestjs-fastify-platform/

├── db/
```

## Architectural Rules & Best Practices

### 1. Strict Dependency Injection

NestJS relies on `reflect-metadata` and TypeScript decorators to resolve dependencies at runtime.
**Crucial Rule:** Never use `import type` for classes injected via a constructor.

```typescript
// ❌ WRONG: TypeScript erases this at compile time. NestJS will fail to resolve it.
import type { CounterService } from "./counter.service.js";

// ✅ CORRECT: Preserves the physical class token for runtime DI injection.
import { CounterService } from "./counter.service.js";
```

### 2. Fastify Body Parser Configuration

Connect-RPC (`@connectrpc/connect-fastify`) requires exclusive control over parsing
`application/json` and `application/connect+json` content types to support streaming and Protobuf
serialization. To prevent Fastify from crashing with a `FST_ERR_CTP_ALREADY_PRESENT` error, the
native NestJS body parser is explicitly disabled in `main.ts`:

```typescript
const app = await NestFactory.create<NestFastifyApplication>(AppModule, fastifyAdapter, {
  bodyParser: false, // Required for Connect-RPC compatibility
});
```

## Running the Server

Ensure you have generated the Protobuf types (`buf generate` in the workspace root) before starting.

```bash
# Install dependencies
bun install

# Start the server (defaults to port 4000)
bun run start
```

### 3. Extending the Application via Inversion of Control (IoC)

When you add a new feature (e.g., `UserService`), you **do not** modify `src/core/connect`.

Instead, follow this pattern:

1. Create `src/features/user/user.rpc-handler.ts` and implement the `RpcHandler` interface.
2. In `user.module.ts`, inject the `RegistryService` from the Core module.
3. Within the `onModuleInit()` hook of `UserModule`, dynamically register the handler:
   `this.registryService.register(UserServiceSchema, this.userHandler)`
4. Import `UserModule` into the root `app.module.ts`.

Because `ConnectModule` utilizes the `onApplicationBootstrap()` lifecycle hook, it will
automatically wait for all features to finish registering themselves before it mounts the Fastify
routes. This achieves true decoupling.
